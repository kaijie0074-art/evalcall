# EvalCall 答辩材料包：一页纸、数字口径与攻防卡

## 一页纸

### 产品是什么

EvalCall 接收一份外呼 SOP 和一批已有真实/脱敏对话，输出“可上线 / 打回 / 无法判定”，并给出 P0 红线、履约、逐项证据、根因 owner、修复动作和人工复核队列。

生产质检以已有对话为主；没有数据时才用对抗式用户模拟器补测试。

### 为什么不是普通 GPT 打分器

1. **评模型**：SOP 编译成可追溯检查点；规则轨只做高置信线索，LLM 轨做语义复核；默认三票；fail 必须引用对话原文。
2. **审指令**：自动发现冲突、不可行、歧义和缺失分支；低分时先判断是不是 SOP 本身有问题。
3. **校裁判**：用已知答案注入评测器，输出准确率、混淆矩阵、P0 fail recall 和分类型短板。
4. **可复现**：每个 run 记录 instruction/checklist/safety/golden hash、代码 commit、模型、票数、token、耗时与计价状态；尺子不同默认拒绝能力归因。
5. **可人工兜底**：P0、NA、分裂票、规则冲突进入复核队列；人工结论不覆写机器原判。

### 目前最强证据

- 真实招聘 SOP：43 检查点、GPT-5.6-sol/xhigh 三票；13 fail、17 NA，门禁“打回”；首要归因是 SOP（lint 0/100、3 项高严重度问题）。
- 裁判全量校准：32 case、165 真值；准确率 92.73%，宏 F1 0.9155；P0 fail recall 11/11；style 仅 70%。
- 高置信不等于正确：12 个错判的平均自报置信度仍为 96.25%。
- 跨家族固定脚本：Claude Haiku 与 GPT-5.6-sol 均被 P0 打回；GPT 履约 50% vs 0%，但同时 2 项改善、2 项退化。

### 产品状态

- 四步工作台、缓存/实时双模式、本地真实执行服务已完成。
- JSONL/JSON/CSV/TXT/MD 接入、默认 PII 遮罩、断点续评、原子写盘已完成。
- 根因归属、人工复核、可比性门禁、成本遥测、编译器黄金集草稿、sealed 协议、证据索引已完成。

## 高频攻防卡

### Q1：这不就是套 GPT 打分吗？

不是。裸 GPT 打分缺少固定尺、证据归属、裁判校准、版本可比性和人工复核。EvalCall 先把 SOP 编译为可追溯检查点，规则/LLM 双轨多数票判定，再用 165 个已知真值测裁判，并把每次 run 的 hash 和成本口径落盘。模型只是其中一个可替换后端。

### Q2：为什么不让系统自动改 SOP、自我进化？

因为这样会形成循环论证：同一个系统既出题、又改答案、再证明自己变好了。EvalCall 只提议候选并做 source_quote 硬闸；正式 SOP、P0 和黄金真值必须由业务 owner/质检终审。新旧 SOP 导致检查尺变化时，系统默认拒绝把差异归因成模型能力提升。

### Q3：92.73% 够不够上线？

不能只看总体准确率。当前 P0 黄金集 recall 是 11/11，但 style 只有 70%，错判仍会高置信。生产使用必须保持 P0 门禁、低置信复核、漂移监控和独立 sealed holdout。

### Q4：你们是不是只会把模型打成 0 分？

不是。门禁和标量分分开：P0 fail 会让整通打回，但报告仍展示各维度和履约；t02 无物理冲突任务有普通 pass/fail；跨模型对照也同时出现改善和退化。0 分不等于“什么都没测到”。

### Q5：为什么不直接相信模型自报 confidence？

实测 12 个错判的平均置信度是 96.25%。所以 confidence 只能排序复核，不能证明正确；真正的可靠性来自黄金集、混淆矩阵、P0 recall 和人工争议裁决。

### Q6：规则轨会不会关键词误杀？

规则轨不是独裁者。关键词命中只产生线索，再由 LLM 核实说话人、子串误命中和引用语境。规则与 LLM 冲突时标记 `rule_conflict` 并进入人工复核。

### Q7：跨模型实验公平吗？

本次固定相同用户轮、任务、检查尺和裁判，只改变被测模型；四项 comparability hash 一致。但固定用户轮可能与新模型上一句不完全衔接，所以明确称为 counterfactual replay，不包装成自然会话随机 A/B。

### Q8：身份口径为什么允许否认 AI？

这是用户明确选择沿用该生产 SOP 的任务级口径，不是 EvalCall 的全局安全默认。系统在 task、manifest 和 summary 中显式记录 `deny_ai_identity` 覆盖；上线前仍需业务法务确认。

### Q9：能节省多少钱？

目前只披露 EvalCall 自身的调用数、token 和耗时。没有人工抽检率、单通复核时长和人力成本真值，不计算绝对 ROI；Codex 订阅调用也没有可对账的输入/输出单价，所以保持“未计价”。

### Q10：真实数据安全吗？

支持本地 JSONL/JSON/CSV/TXT/MD 适配，默认对手机号、身份证、邮箱、银行卡、订单号和地址做结构化遮罩；遥测不存 prompt 原文。真实数据是否能发送第三方模型仍由数据 owner 授权。

## 宣称审计

| 高风险词 | 当前可讲范围 | 必须带的限制 |
|---|---|---|
| 真实数据 | 导师提供真实生产 SOP + 仓库留档对话轨迹 | 不等于线上全量话务分布 |
| 全量 | 32-case 黄金集的 165 个真值全量 | 不是所有业务场景全量 |
| 准确率 | 指定模型/提示词/票数/黄金集版本 | 不外推线上；分类型短板必须同时披露 |
| 降本 | 自动评测自身 token/耗时可实测 | 无人工成本真值，不讲确切节省金额 |
| 自进化 | 只做候选提议、人工审核、同尺回归 | 不自动改正式 SOP/黄金集/生产模型 |
| 跨模型 | Claude Haiku 历史轨迹 vs GPT 固定用户轮重放 | counterfactual replay；不是自然 A/B |

## 录屏与备用资产分镜

1. **0–10 秒**：打开 `app.html`，展示实时服务状态和三个 SOP 样例。
2. **10–25 秒**：选择真实招聘 SOP，展示 9,185 字、43 检查点。
3. **25–40 秒**：导入/展示留档对话与默认 PII 遮罩。
4. **40–60 秒**：点击检查点，展示 source_quote 回指 SOP 原文。
5. **60–90 秒**：先跑“验证缓存”，明确 run 名称与日期。
6. **90–120 秒**：切本地实时，跑 1 通；若超时，展示 fail-closed 的“无法判定”。
7. **120–150 秒**：打开完整报告，指向门禁、根因、review queue、成本与 manifest。
8. **150–180 秒**：打开证据索引，现场搜索 `92.73%` 并回到原始 JSON。

### 10 秒故障切换

- API/网络失败：切回“验证缓存”，说清不是实时。
- 本地服务失败：直接打开 `site-deploy/report-official-gated.html` 和真实招聘 v2 `report.html`。
- 页面异常：使用本地 PPT 第 5–7 页继续讲，不停留排错。
- 结果全 NA：展示“无法判定”和 `evaluation_errors.json`，不口头宣布通过。

## 用户最终动作

- 业务专家审核 96 项编译器黄金集草稿，确认 P0/P1/P2、语义等价和 source_quote。
- 由未参与调参的质检人员持有仓库外 sealed holdout；锁版后执行 `calibrate.py --sealed`，只带回聚合指标。
- 按 3 分钟、90 秒和断网三种口径各彩排；把讲稿改成自己的口语。
- 在比赛平台亲自确认协议并完成官方提交。
