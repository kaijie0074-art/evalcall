# EvalCall 答辩演示脚本（3 分钟）

> 对应：`EvalCall答辩稿-可信评测基础设施-20260712.pptx`。现场以产品工作台为主，PPT 只负责建立可信度。

## 0:00–0:20｜开场：一句话说清产品

大家好，我们做的是 EvalCall。

上传一份外呼 SOP 和已有对话，它不只给分，而是直接告诉你：这批模型可上线、应打回，还是证据不足无法判定；如果有问题，还会告诉你该改模型、SOP、裁判还是测试数据。

## 0:20–0:42｜为什么普通打分器不够

外呼评测真正难的不是打一个分，而是证明这个分可信。

一个 0 分可能有四种原因：模型没守指令、SOP 自相矛盾、裁判误判，或者测试根本没覆盖真实拒绝场景。任何一层出错，结论都会误导上线决策。

## 0:42–1:08｜现场工作台：四步主链

现在看产品。

第一步选 SOP；第二步导入真实或脱敏对话；第三步确认 L0 通用红线和 L1 业务检查尺，每一条都能点回 SOP 原文；第四步执行评测。

输出不是一个总分，而是门禁、P0、履约、逐项证据、根因 owner 和人工复核队列。

现场建议选择“骑手招聘外呼”，先展示缓存，再切本地实时跑 1 通。强调：缓存和实时结果明确分标，单通实时只用于验证链路，不替代正式三票批次。

## 1:08–1:32｜三层闭环：为什么可信

我们用三层闭环回答三个问题。

第一层评模型：指令编译、对抗模拟、规则与 LLM 双轨、默认三票，每条 fail 都引用对话原文。

第二层审指令：自动检查冲突、不可行、歧义和缺失分支。先治指令，再评模型。

第三层校裁判：把已知答案注入评测器，输出混淆矩阵和 P0 查全率，把裁判自己的准确率印在报告上。

## 1:32–1:58｜真实招聘 SOP：系统敢于打回

这份导师提供的真实招聘 SOP 有 43 个检查点。

我们用 GPT-5.6-sol、xhigh、三票评测留档对话，得到 13 fail、17 NA，门禁是“打回”。

但系统没有简单归责外呼模型：指令体检是 0 分，并发现 3 项高严重度问题，所以首要 owner 是 SOP 业务方；模型问题只给低置信归因。动作是先修 SOP，再同尺回归。

## 1:58–2:25｜165 个真值：裁判也要考试

我们让裁判做了 32 case、165 个已知真值的全量考试。

总体准确率 92.73%，P0 fail 在这份黄金集中 11 个全检出；constraint 和 forbidden 都是 100%。

但 style 只有 70%，而且错判时平均自报置信度仍有 96.25%。所以模型说“我很确定”不等于真的可靠，黄金集和人工复核不能省。

## 2:25–2:45｜跨模型：不讲“全面胜出”

我们还固定了同样的 20 个用户轮，对 Claude Haiku 和 GPT-5.6 做同尺对照。

GPT 的履约从 0% 到 50%，违规率略降，但同时有 2 项改善和 2 项退化；两边都触发 P0 打回。

我们的结论不是“更大的模型一定更好”，而是每个模型在具体业务约束上都有可操作差异。

## 2:45–3:00｜收口：评委应该记住什么

EvalCall 交付的不是一个分数，而是一条可审计的上线决策链。

每个数字都能在 30 秒内回到 run、JSON 指针和文件 hash；每次迭代都能同尺比较，低置信项自动交人。

接下来请看现场实跑：选 SOP、导入对话、确认检查尺，然后打开完整证据报告。

## 现场故障切换

- 实时服务不可用：10 秒内切回“验证缓存”，明确说“这是标注日期和 run 的预跑结果”。
- 模型超时：展示 `evaluation_errors.json` 与 fail-closed 的“无法判定”，不要口头宣称通过。
- 评委追问准确率：回答“32 case、165 真值、单模型 1 票；总体 92.73%，P0 11/11，但 style 仅 70%，不外推线上分布”。
- 评委追问身份合规：本次生产 SOP 已按用户选择记录 `deny_ai_identity` 任务级口径，并在 manifest 中显式覆盖；该口径需业务法务最终确认，不作为全局默认。
