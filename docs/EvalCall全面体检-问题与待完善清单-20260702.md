## 一、P0 · 现场必炸（1 条）

### 1. `grow` 子命令 100% 崩溃
- `evalcall/grow.py:71` 调 `llm.chat_json(messages, model=model)` 漏传必填参数 `schema_hint`（签名见 `llm.py:375`）。**实跑验证恒报错**，`python -m evalcall grow` 必然 `_die`。
- README 把它写在卖点命令区，评委照抄就炸。修复只需一行。
- 测试没抓到的原因：单测只测了 `filter_traceable` 纯函数，stub 的 `chat_json` 签名比真实版宽松；`runs/` 下没有任何一次真实 grow 产物——这条链路**从未端到端跑通过**。

## 二、P1 · 卖点级缺陷（评委细看就露馅，4 条）

### 2. run_id 数据流断裂 → "证据跳转"在真实 run 上永远失效
- `arena.py:177` 生成 uuid 型 run_id；`cli.py:228` 另造 `{task}__{persona}__{i}` 复合 ID；`cli.py:250` 的 `setdefault` 无效（arena 已填值，覆盖不了）。
- 真实产物实证：`runs/official01_n3/` 里 judgments 与 transcripts 两套 ID 完全对不上。报告模板的 `#call-{run_id}` 锚点因此永不匹配。
- 演示截图看不出问题是因为 `runs/_demo` 是手工写的一致 ID。修法一行：落盘前强制 `trajectory["run_id"] = run_id`。

### 3. 失败案例回放把多条轨迹的证据混到一条对话上
- `report.py:161` 按 `(task, persona)` 索引轨迹，n=3 时互相覆盖只剩最后一条；`report.py:564-588` 又按同键聚合违规轮次。
- 后果：轨迹 1 第 4 轮的违规被高亮到轨迹 3 第 4 轮上——**评委细看回放时"标红的那句话"可能完全没违规**。聚合键应改为 run_id（依赖第 2 条先修）。

### 4. 判定通道故障时门禁 fail-open：全 na → 打出"可上线"
- `judge.py:210-217` LLM 批判定异常时整批退化 na；`judge.py:408-435` na 不进分母 → 无 fail → `gate="可上线"`、平均分 0.0，自相矛盾。
- 叠加：规则轨置信 1.0 命中的禁语，只要 LLM 轨因超时/网络挂了返回 na，确凿证据被静默丢弃。"LLM 不可用"和"LLM 否决"被混为一谈。
- 建议：na 占比超阈值时 gate 输出「无法判定/需重跑」。

### 5. `--checklist` 同尺复用通道剥掉 `safety`/`policy_source`，安全红线降级
- `cli.py:126-128` 重建 Checkpoint 白名单漏了这两个字段（`cmd_grow` 的白名单却有——是遗漏不是设计）；`cli.py:146` 去重又把带 safety 的原版挡掉。
- 今天没爆只因红线全是 critical 兜底；一旦出现 `severity: major, safety: true` 的红线，A/B 同尺对比（文档强推的主路径）会把它漏出门禁。
