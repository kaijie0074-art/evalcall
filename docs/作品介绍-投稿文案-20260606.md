# EvalCall — 外呼对话模型指令遵循自动评测系统

> 三层质量闭环：评模型 · 审指令 · 校裁判
> 在线演示：https://kaijie0074-art.github.io/evalcall/ ｜ 代码：https://github.com/kaijie0074-art/evalcall

## 项目背景与痛点

美团履约数字人每天产生大规模外呼对话。任务指令往往包含复杂的流程分支和多重合规约束——关键信息播报顺序、禁止话术、隐私保护要求、情绪处理规范等。当前质检高度依赖人工抽听：全量覆盖成本不可承受，问题发现滞后无法支撑快速迭代，评分标准因人而异难以追溯。

但「做一个自动评测器」不是这道题的难点——**难的是凭什么信它**：LLM 裁判自己会误判，谁来评测评测系统？模型分数低，是模型差还是指令本身写得自相矛盾？模拟器说「测得充分」，充分到什么程度？

EvalCall 的回答：**三层质量闭环——不仅评测模型，还审计指令本身，并量化校准自己的裁判。** 每一层都在官方脱敏数据上实测落地。

---

## 创新性：三层质量闭环

**第一层 · 评模型：指令编译 + 对抗模拟 + 双轨判定。**
把自然语言任务指令编译成结构化检查点清单，每条标注类型（流程/硬约束/禁止项/话术）与 `source_quote`（可溯源指令原文）。六种 persona（配合/打断/跑题/质疑/情绪/沉默）的对抗式模拟器定向诱导被测模型违规；规则轨处理确定性判断，LLM 轨携带指令原文逐条判定并引用对话原文为证，3 票多数决定结论。评测结论不是「大模型觉得还行」，而是逐条可审计的通过/失败。

**第二层 · 审指令：Instruction Lint，评测的不只是模型。**
指令体检器自动检测任务指令的自相矛盾、不可行约束、歧义表述、缺失分支，并给出可直接照改的修订建议。**官方数据实测战绩（全部为系统自主发现）**：
- 任务二「每次回复最多15-20字」与官方参考话术（单句约43字）物理冲突——任何模型都无法同时满足，实测被一票否决归零，根因在指令不在模型；
- 任务一 Call Flow 与 Knowledge Points 对「单日合同履约周期」的定义互相矛盾，模型说哪个都算错；
- 任务二全流程缺失「商家明确拒绝/请勿再来电」退出分支——直接的合规风险；
- 任务一占位符 X/Y/Z/W 未注入且同名变量两种语义。

对每天编写外呼指令的运营团队，这是上线前的体检环节：**先治指令，再评模型，质检数据才有意义。**

**第三层 · 校裁判：已知答案注入，评测结果自带误差棒。**
32 例合成对话黄金集（165 个已知真值，含引用用户脏话的关键词陷阱、违规后自纠、方言+ASR噪声多违规并发等边界情形），跑完整裁判流程输出查准率/查全率/混淆矩阵——**我们把自己裁判的准确率印在报告上**。关键实验：同模型连投 3 票会 3:0 一致地错（只消方差不消偏见），换成 haiku/sonnet/opus 跨模型裁判团后准确率 86.5%→94.2%、违规误报近半减、漏判保持为零。配套：裁判提示词携带指令原文、被测/模拟器上下文隔离、鲁棒 JSON 解析、按模型分别置信度校准（跨模型置信度不在同一刻度，已主动披露并修复）。

**测试充分性量化：coverage-guided 反馈环（实证有效）。**
每条轨迹判定后，未触达检查点自动成为下一条轨迹的优先攻击目标。实测：被动统计模式下任务二有 6 个条件分支永远测不到（触达率 77.8%），开启反馈环后两任务触达率均达 **100%、盲区清零**——发现盲区到消灭盲区闭环，对标 FLARE 的 coverage-guided behavioral fuzzing，首次用于对话指令遵循质检。

## 完整性

系统完整交付赛题要求的两个核心目标，并超出交付：

- **用户模拟器**：多 persona × 多策略 × 对抗模式 + coverage-guided 反馈环（触达率实测 100%），测试充分性可证明。
- **评测报告**：逐检查点明细（结论 + 证据引用 + 置信度 + 3票投票详情）、四维雷达图、失败案例完整回放、persona 维度切片、覆盖率与盲区、裁判可靠性度量，过程可解释、结果可量化。
- **超出交付**：指令体检报告（含修订建议）、裁判校准报告（查准率/查全率/混淆矩阵）、从既有轨迹全量重判工具（裁判升级后无需重跑对话）。

端到端流程闭环：任务 YAML → 指令编译 → 多轨迹模拟 → 批量判定 → HTML/JSON 报告。官方脱敏数据零改造直接接入（格式适配层解耦）。

## 应用效果

| 指标 | 说明 |
|------|------|
| 约束违反数/百次对话 | 核心量化产出，可按 task / persona / severity 三维切片 |
| 检查点触达率 + 盲区清单 | 测试充分性的量化表达，反哺模拟器调度 |
| 裁判查准率/查全率/混淆矩阵 | 已知答案注入校准，评测结果自带误差棒 |
| 指令可行性分 + 缺陷清单 | 上线前拦截指令缺陷，附可照改的修订建议 |

**开发过程中系统抓出的真实问题**：官方指令的物理冲突与业务矛盾（第二层）、裁判对「指定开场白」类检查点的系统性误判（第三层）、8 个从未被测试触达的检查点分支（覆盖率）——三层闭环不是概念，每一层都有实测战绩。

## 商业价值

**一、质检自动化降本。** 外呼规模越大，人工质检的成本优势越显著。系统支持全量覆盖，消除抽样盲区，问题可在上线前批量发现。

**二、指令资产质量管控。** 指令体检把「指令写得好不好」变成可量化、可拦截的上线前检查——这是外呼业务每天都在生产的资产，体检器直接提升整条业务线的指令质量基线。

**三、模型迭代加速。** 每次模型版本更新，一次评测跑出量化对比报告；裁判升级后用重判工具从既有轨迹秒级回溯，无需重跑对话。

**四、横向复用。** 指令编译器、双轨判定引擎、体检器、校准器均为通用组件，客服外呼、营销外呼、预约提醒等场景只需更换任务 YAML。

## 判别力与选型实证（受控实验，同一固化尺子）

**实验一 · 约束消融**：同模型同任务，唯一变量是被测模型「看得见 vs 看不见」指令约束。结果四维画像完美分离（约束遵循 100% vs 28.6%、话术合规 62.5% vs 0%），同时实证「标量总分会误导，分维度画像才可靠」——守约束的模型因流程信息量被字数挤压反而总分更低，单看分数会得出荒谬结论。

**实验二 · 真实模型对比**：Sonnet vs Haiku 当被测模型。反直觉发现：**小模型 Haiku 更守指令**（约束遵循 100% vs 66.7%、话术合规 87.5% vs 50%）——强字数约束场景下，输出天然简短的模型更贴合极简话术要求。**这正是外呼模型选型不能看通用能力榜单、必须做指令遵循实测的直接证据**，对美团选型决策有直接价值。

**统计口径**：官方两任务每 persona 3 条独立轨迹，报告均值±标准差与 pass^k 可靠性指标（τ-bench 口径）；两任务 pass^3=0 的根因经指令体检归因于指令缺陷而非模型能力，报告自动挂归因警示。
