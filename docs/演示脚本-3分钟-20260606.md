# EvalCall 演示脚本 — 3 分钟

---

## 00:00–00:30 | 痛点（30秒）

**[口述]**

美团每天有大量外呼对话——催单确认、配送预约、超时安抚。这些对话有严格的指令约束：必须按顺序播报关键信息、绝对不能说某些话、碰到情绪用户要有特定处理流程。

问题是：**怎么知道模型有没有按指令来？**

现在靠人工抽听——贵、慢、覆盖率低、评分标准因人而异、出了问题只能靠投诉发现。

这就是 EvalCall 要解决的问题：把"人工看有没有按指令来"这件事，变成可量化、可重复、有证据的自动化评测。

---

## 00:30–01:30 | 跑一次评测实况（60秒）

**[切换到终端]**

先看一下任务指令长什么样：

```bash
cat data/tasks/delivery_confirm.yaml
```

**[口述]** 这是一个外卖催单确认任务——有流程节点、硬约束、禁止话术，普通的任务指令就是这个样子。

现在跑一次评测：

```bash
export EVALCALL_BACKEND=claude-cli
python -m evalcall run --task data/tasks/delivery_confirm.yaml --personas all --runs 3
```

**[口述，对着终端输出]**

系统在做三件事：

第一，指令编译器把这份 YAML 编译成检查点清单——每条检查点对应指令里的一个约束，有 type、severity，还有 source_quote 指向原文。

第二，用户模拟器开始工作。这里能看到 persona：配合型用户、打断型用户、情绪型用户，三条轨迹并发跑。对抗模式下，模拟器会主动往约束的"边界"去踩，看被测模型会不会违规。

第三，双轨评测引擎在逐条判定——规则轨直接检查关键词，LLM轨逐检查点给出 pass/fail，3票投票定结论。

**[等待跑完，指向终端摘要行]**

三条轨迹跑完，总分和约束违反数已经打出来了。下面看完整报告。

---

## 01:30–02:30 | 报告讲解（60秒）

**[打开 runs/ 目录下的 report.html]**

```bash
open runs/*/report.html
```

**[口述，指着报告各区域]**

顶部是总分和四维雷达图——流程完整度、约束遵循率、异常处理、话术合规，一眼看出哪个维度有问题。

往下是**约束违反数/百次对话**——这是我们的核心量化指标，按 severity 分级，critical 单独列出来。

**[点到某条 fail 的检查点明细]**

这条检查点 fail 了——看这里：结论是"未按顺序播报订单金额"，置信度 0.92，然后是具体证据：

> 第 4 轮，用户模型说了"……配送费 3 元……"但在此之前没有先说订单总金额。引用原文在这里。

这是系统强制要求的：**每条 fail 必须有对话原文引用**，不允许无根据的结论。质检员不用重听录音，看报告直接定位问题。

最后是 persona 维度切片——情绪型用户下约束违反率明显高于配合型用户，说明被测模型在情绪场景下处理有缺口，这是下一轮迭代的方向。

## 02:30–03:00 | 商业价值收尾（30秒）

**[口述]**

总结一下 EvalCall 的价值：

对美团来说，外呼规模越大，人工质检成本越高，覆盖率越低。EvalCall 实现全量覆盖，成本随并发线性可控，问题在上线前就能批量发现，而不是等投诉。

对研发团队来说，每次模型迭代后一次 `evalcall run` 出一份对比报告，约束遵循率的变化趋势一目了然，不需要等评估窗口。

而且这个框架不绑定外卖催单——换一个 YAML，客服外呼、营销外呼都能直接复用，评测引擎一行代码不用改。

## 备用应对（如被追问）

**Q：检查点编译准确吗，如果编译错了怎么办？**
A：检查点清单会在评测前输出给用户审查，每条都有 source_quote 指向原文，人工可以一分钟内扫完确认。这比事后争"为什么判 fail"要高效得多。

**Q：LLM-as-Judge 自己会不会也出错？**
A：系统输出 judge 自一致率和双轨冲突率作为可靠性指标，冲突的判定结果会在报告里高亮标出，让人工二次复核；3票多数投票也降低了单次误判对结论的影响。

**Q：真实数据接入复杂吗？**
A：有格式适配层，字段映射表在 `data/README.md`。脱敏轨迹放进指定目录，直接跑 `evalcall report`，不需要动评测逻辑。
