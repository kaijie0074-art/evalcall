# EvalCall 下一阶段总建模（AI 执行面 × 人工决策面）

> 项目：EvalCall 外呼对话模型指令遵循自动评测系统（美团黑客松赛道二）
> 日期：2026-07-12
> 输入：2026-07-12 导师录音文字稿 + 当前代码/测试/运行产物 + 2026-06-30 与 2026-07-02 两轮已完成建模
> 本文用途：唯一的下一阶段执行蓝图。已完成能力只作基线，不重复立项。
> 默认授权边界：本文是建模，不等于已授权 push、部署、花费 API 额度、提交比赛或处理未脱敏业务数据。

---

## 0. 一句话目标

把 EvalCall 从“已经很强的评测方法与报告生成器”收口成“真实用户能操作的外呼质检产品”：

> 输入一份 SOP/系统指令和一批真实或脱敏对话，自动生成可溯源检查点，逐通评测，输出可上线/打回/无法判定，并说清该改外呼模型、SOP、裁判还是测试数据。

产品主链：

```text
SOP + 已有对话
  → 输入校验/脱敏
  → L0 通用红线 + L1 SOP 业务检查点
  → 人工确认检查尺
  → 逐通双轨判定
  → 批量聚合/上线门禁
  → 根因归属/行动建议
  → 人工复核/回归验证
```

“评模型·审指令·校裁判”保留为可信度的三层保证，不再展示为用户必须顺序操作的三个阶段。对抗模拟是“没有真实对话时的测试数据生成器”，不是生产质检的必经入口。

## 1. 现状基线（不重复做）

以下能力已在代码或真实产物中落地，新建模只做接线/产品化，不另起炉灶：

- 指令→可溯源 checklist 编译。
- 安全/合规红线轨与 P0/P1/P2 分级。
- 对抗 persona、coverage-guided 反馈环。
- 规则 + LLM 双轨裁判、3 票多数决、分歧复核。
- 履约 outcome、AI 身份质疑应对、P0 上线门禁。
- 通话级清单、证据跳转、弱点聚合、热力图、盲区清单。
- lint 指令体检、报告低分归因警示。
- judge 黄金集/held-out 校准、混淆矩阵、P0 查全率。
- `diff` 跨版本回归、`grow` 可溯源活清单候选。
- 真实官方任务 run、真实生产 SOP lint、报告和线上展示页。
- 70 个测试当前全绿。

## 2. 设计红线（任一节点违反即回退）

- **R1 溯源红线**：L1 检查点必须回指 SOP 原文；L0 必须回指显式 policy 来源。
- **R2 固定尺红线**：模型对比、版本回归必须复用同一 checklist 及安全 policy 版本。
- **R3 反循环论证**：对话数据只能提议“SOP 已有原文但清单漏掉”的候选，不得从被测模型输出发明新标准。
- **R4 不伪量化**：实测、估算、业务假设必须分栏；没有人工成本真值时不声称确切 ROI。
- **R5 失败关闭**：裁判失败、输入不完整、元数据不可比时不得输出“可上线”或“模型变好”。
- **R6 人机分权**：Agent 可以生成建议、候选和预标注；业务口径、合规冲突、真值、上线与外发由人拍板。
- **R7 私隐边界**：未脱敏生产对话不进入不明确的第三方后端；输出报告不泄露电话、地址、订单号等个人信息。

## 3. 责任分类

| 类型 | 含义 | 执行规则 |
|---|---|---|
| **A：Agent 可独立执行** | 本地代码、测试、静态页、文档、已授权数据上的实验 | 用户授权进入实施后，Agent 可按依赖持续执行至验收 |
| **H：协作节点** | Agent 能做主体，但需用户提供权限、数据、预算或业务口径 | 遇到必需条件时停在该节点，不擅自假设 |
| **U：用户专属节点** | 真值裁决、未公开 holdout、演讲、官方提交、业务授权 | Agent 准备材料/选项，用户做最终动作 |
| **F：未来项** | 超出当前黑客松产品闭环 | 只进路线图，默认不实施 |

## 4. Agent 可执行的完整节点模型

### M0 · 基线锁定与防重复

- [ ] **A0-1 锁定当前真实基线**
  - 动作：记录 git commit、测试数、已有 run、已有线上页面和未提交用户改动。
  - 判据：生成基线快照；开发不覆盖用户已有未提交文件。
  - 依赖：无。
- [ ] **A0-2 宣称→代码→产物对照表**
  - 动作：将 README/首页/PPT 中的主张对应到代码入口和真实 run，标记已实测/估算/路线图。
  - 判据：所有核心动词都有代码或降级表述；无“文档说可用但实际不通”。
  - 依赖：A0-1。

### M1 · 真实对话离线评测主链（最高优先级）

- [ ] **A1-1 统一外部对话 schema**
  - 动作：定义 SOP/task 与 transcript 的必填、可选、禁止字段；明确 `run_id/task_id/persona_id/turn/role/content/meta`。
  - 判据：一份机器可读 schema + 一份人类示例；缺字段时给出带行号的明确错误。
  - 依赖：A0-1。
- [ ] **A1-2 输入校验与规范化层**
  - 动作：检测空对话、重复 ID、非法 role、非单调 turn、超长对话、缺 task_id；对可修复项生成 normalization report。
  - 判据：坏输入不能静默进入 judge；可修复输入的修复行为可追溯。
  - 依赖：A1-1。
- [ ] **A1-3 新增 `evalcall evaluate` 主命令**
  - 动作：接受 `--task/--transcripts/--checklist/--out/--votes/--model`，跳过 simulator，直接对已有对话逐通判定和聚合。
  - 判据：新建只含 task + transcripts 的目录，一条命令产出 checklist/judgments/judgments_by_run/summary/report；无需预存 judgments。
  - 依赖：A1-1、A1-2。
- [ ] **A1-4 批次容错与断点续评**
  - 动作：单通失败记录 error 而不丢整批；支持跳过已完成 run_id；保持中间产物原子化。
  - 判据：人为中断后重跑不重复消耗已完成调用；失败通话可单独重试。
  - 依赖：A1-3。
- [ ] **A1-5 报告入口 fail-closed**
  - 动作：`report` 发现只有 transcripts 但没有 judgments 时，给出“请先 evaluate”的显式错误，不生成空报告。
  - 判据：缺 judgments 用例返回非 0 且提示正确命令。
  - 依赖：A1-3。
- [ ] **A1-6 真实数据格式适配器**
  - 动作：支持 JSONL 主格式；增 CSV/纯文本/ASR 时间戳格式的可选转换器。
  - 判据：每类至少一个 fixture 能转成同一内部 schema；转换不丢 speaker/时间戳/原始 ID。
  - 依赖：A1-1。
- [ ] **A1-7 脱敏检测和报告红线**
  - 动作：对手机号、订单号、地址、身份证、姓名等高风险字段做本地检测；默认报告中遮罩。
  - 判据：合成 PII 样例不会在 HTML/JSON 对外版中原样出现；保留内部追溯 ID。
  - 依赖：A1-2。
- [ ] **A1-8 主链回归测试与文档对齐**
  - 动作：新增 evaluate 的 unit/integration/smoke 测试；修 README 脱敏数据接入章节。
  - 判据：所有 README 命令可复制运行；原 70 测试不回归；新主链至少有一条真实签名的端到端测试。
  - 依赖：A1-3～A1-7。

### M2 · 运行可复现、成本与可比性

- [ ] **A2-1 Run Manifest 四元组扩展**
  - 动作：为每次 run 记录 instruction/checklist/golden/policy hash、target/judge model、votes、seed、commit、时间、backend。
  - 判据：`summary.json.meta` 字段齐全；同文件重跑 hash 一致，文件变更 hash 必变。
  - 依赖：A1-3。
- [ ] **A2-2 投票配置单一口径**
  - 动作：统一 CLI 默认、`N_VOTES`、函数默认与 README/GOVERNANCE；明确 `--votes` 与环境变量优先级。
  - 判据：命令行、直接函数和文档对同一输入得到同一票数；测试覆盖优先级。
  - 依赖：无。
- [ ] **A2-3 LLM 调用观测层**
  - 动作：统一记录每次调用所属阶段、模型、耗时、成功/失败、重试、输入/输出 token（后端不提供时标 estimated）。
  - 判据：每个 run 生成 telemetry JSON；任意报告的调用数与 telemetry 一致。
  - 依赖：A1-3。
- [ ] **A2-4 价格配置与成本计算**
  - 动作：价格作为带日期/来源的外部配置；分离实测 token、估算 token、估算金额。
  - 判据：单通/整批/分阶段成本可对账；无价格时显示“未配置”而非 0 元。
  - 依赖：A2-3；H2 提供可用的计价口径。
- [ ] **A2-5 可比性门禁**
  - 动作：`diff` 比较前检查 instruction/checklist/policy/golden 元数据；尺子不同时停止或标红“不可归因”。
  - 判据：同尺 diff 正常；不同尺不能被系统自动表述为模型改进/退化。
  - 依赖：A2-1。
- [ ] **A2-6 成本/时延报告区块**
  - 动作：报告展示调用数、token、耗时、单通成本与口径声明。
  - 判据：HTML 数字与 telemetry 和价格配置精确对账。
  - 依赖：A2-3、A2-4。

### M3 · 统一根因归属与行动建议

- [ ] **A3-1 归因信号 schema**
  - 动作：将 lint 冲突、检查点失败率、模型对比、judge 分歧、规则冲突、校准质量、persona 集中度、NA 率统一为可审计信号。
  - 判据：每个归因都能回指原始数字或证据，不只有 LLM 文字。
  - 依赖：A1-3、A2-1。
- [ ] **A3-2 四类根因规则引擎**
  - 动作：输出 `target_model / instruction / judge / test_data / uncertain`；必要时允许多根因并存。
  - 判据：至少 5 类合成场景可稳定触发预期归因；证据不足输出 uncertain，不强行归类。
  - 依赖：A3-1。
- [ ] **A3-3 建议动作库**
  - 动作：将根因映射到可操作动作：修 SOP、补分支、重判、人工复核、补真值、补对话分布、回归重测。
  - 判据：建议不直接改正式 SOP/checklist/judge prompt；每条标出 owner 和验证方式。
  - 依赖：A3-2。
- [ ] **A3-4 报告“该改哪里”决策卡**
  - 动作：在门禁头条下方展示首要根因、次要根因、证据、置信度、建议 owner/下一步。
  - 判据：评委不展开长表格也能在首屏回答“为什么打回、谁来改”。
  - 依赖：A3-2、A3-3。
- [ ] **A3-5 人工复核导出/回填协议**
  - 动作：导出需复核项 JSON/CSV，接受人工结论回填；保留机器原判与人工结论。
  - 判据：人工结论不覆写原记录；重新聚合可生成“机器版/人工终审版”两种口径。
  - 依赖：A3-2；U5 提供真实复核口径。
- [ ] **A3-6 归因回归测试**
  - 动作：用指令冲突、裁判分歧、模型高频失败、极端 persona 集中、证据不足五类 fixture 锁定规则。
  - 判据：修改归因阈值必须显式更新测试与口径记录。
  - 依赖：A3-2～A3-5。

### M4 · 可演示的产品界面

- [ ] **A4-1 信息架构重排**
  - 动作：首屏只讲输入、处理、输出；三层闭环移入“为什么可信”；学术对标/长数据下沉。
  - 判据：新访客 10 秒内能说出“两个输入 + 一个决策输出”。
  - 依赖：A0-2。
- [ ] **A4-2 四步式操作流**
  - 动作：任务选择/上传 → 对话选择/上传 → 检查尺确认 → 评测结果；演示模式可用内置样例。
  - 判据：页面上时刻可见当前步骤、输入摘要、下一步行为；无必须读 README 才会用的步骤。
  - 依赖：A1-3、A4-1。
- [ ] **A4-3 SOP ↔ 检查点可视化**
  - 动作：左侧 SOP 原文，右侧 L0/L1 检查点；点击检查点高亮 source_quote；无源候选显式拦截。
  - 判据：任意正式检查点一次点击可看到来源；L0 能看到 policy 来源。
  - 依赖：A4-2。
- [ ] **A4-4 进度与可恢复状态**
  - 动作：展示校验、编译、判定 n/N、聚合、渲染；错误可定位到单通并重试。
  - 判据：运行超过 10 秒时用户不会看到静止空白页；中断后页面能说明已完成多少。
  - 依赖：A1-4、A4-2。
- [ ] **A4-5 结果首屏决策化**
  - 动作：首屏显示门禁、履约率、P0、复核项、最高频失败、根因和 owner；长明细折叠。
  - 判据：不滚动页面即可回答“能否上线、为什么、谁来改”。
  - 依赖：A3-4。
- [ ] **A4-6 现场模式 + 缓存模式**
  - 动作：支持 1～2 通真实现跑；大批次使用预跑且显式标注的缓存产物；不冒充实时。
  - 判据：无网络/超时时可在 10 秒内切换到缓存报告；缓存页面显示 run 时间与模型。
  - 依赖：A2-1、A4-2。
- [ ] **A4-7 视觉/响应式/可访问性验货**
  - 动作：检查桌面端、笔记本屏幕、移动端；校验对比度、键盘操作、长文本溢出。
  - 判据：核心四步页面均有截图验收；无遮挡、横向溢出和不可点按钮。
  - 依赖：A4-2～A4-6。

### M5 · 证据包与实验

- [ ] **A5-1 三类 SOP 演示集合**
  - 动作：官方任务一、官方任务二、真实生产 SOP、无冲突对照任务中至少选 3 类；统一产物口径。
  - 判据：至少同时出现 pass、普通 fail、P0、复核、SOP 冲突五类结果；不手改 judgments 凑故事。
  - 依赖：A1-3。
- [ ] **H5-2 完成真实生产 SOP 的剩余评测**
  - 动作：从已留档的 43 个检查点 + 1 轨迹续跑，或用新 evaluate 评已有真实/脱敏对话。
  - 判据：生成一份非手造、可审计的完整报告；如指令存在“否认 AI 身份”冲突，按 U6 拍板口径处理。
  - 依赖：A1-3；H1 数据授权；H2 模型访问/预算；U6 合规口径。
- [ ] **H5-3 跨模型家族同尺对照**
  - 动作：至少选 Claude + DeepSeek/Qwen/其他非 Anthropic 家族；固定对话/checklist/judge 口径。
  - 判据：输出门禁、履约、P0、检查点 diff、成本、耗时；manifest 证明只变一个主变量。
  - 依赖：A2-1、A2-5；H2 凭据/额度；U2 决定模型集合。
- [ ] **A5-4 实测成本证据卡**
  - 动作：从 telemetry 生成每通调用数/token/时间/成本；与已有静态估算对账。
  - 判据：估算与实测差异被说明；不得用无来源人工成本计算确切节省金额。
  - 依赖：A2-6、H3（如需人工对比）。
- [ ] **H5-5 编译器黄金集草稿**
  - 动作：Agent 为多份 SOP 生成人工待审的标准检查点、类型、严重度和 source_quote；用户/业务专家终审。
  - 判据：能计算检查点召回率、无源率、类型准确率、severity 准确率。
  - 依赖：U4 完成业务真值裁决。
- [ ] **U5-6 Sealed holdout 与 canary 协议**
  - 动作：Agent 只提供格式、抽样和评分脚本；未公开案例由用户/质检负责人保管，不给调参者看。
  - 判据：holdout 案例未进入公开仓库；评测脚本只输出统计而不泄露真值。
  - 依赖：U4、U5。
- [ ] **A5-7 证据包自动索引**
  - 动作：将每个主张映射到 run/报告/测试/截图/口径。
  - 判据：答辩数字能在 30 秒内定位原始产物；同一数字在各文档中无冲突。
  - 依赖：A5-1～A5-5。

### M6 · 答辩、文档与发布收口

- [ ] **A6-1 README 重写产品入口**
  - 动作：开头优先展示 SOP + 对话 → 上线决策；给出 simulate/evaluate/report 三类不同入口。
  - 判据：新用户可不阅读架构章节就跑通真实对话评测。
  - 依赖：A1-8。
- [ ] **A6-2 三分钟演示脚本重写**
  - 动作：按“输入→检查尺→现跑 1～2 通→门禁→证据→归因→成本”排练；三层闭环只做可信度收口。
  - 判据：口播常速不超 3 分钟；每句数字可回指证据包。
  - 依赖：A4-6、A5-7。
- [ ] **A6-3 PPT/一页纸/攻防卡对齐**
  - 动作：统一产品主线、真实数字、已实现/路线图标识；补“为什么不是裸 GPT 打分”“为什么不自动改 SOP”。
  - 判据：三份材料数字口径一致；已知局限主动披露。
  - 依赖：A5-7。
- [ ] **A6-4 演示全链路录屏脚本与备用资产**
  - 动作：准备无网络缓存报告、关键截图、完整录屏分镜、演示数据副本。
  - 判据：现跑失败后 10 秒内无痕切换备用资产；备用资产与当前代码版本一致。
  - 依赖：A4-6、A6-2。
- [ ] **A6-5 发布前宣称审计**
  - 动作：对“真实数据”“全量”“准确率”“降本”“自进化”“跨模型”等高风险宣称逐条审核。
  - 判据：每条标记实测范围、样本量、证据和限制；无超出产物的表述。
  - 依赖：A0-2、A5-7。
- [ ] **H6-6 本地收尾、Git 与部署**

## 5. 需要用户提供条件的协作节点（H）

### H1 · 数据与隐私授权

用户需确认：

- 哪些真实对话/SOP 可在本地处理。
- 是否已经脱敏，可否进入第三方 LLM API。
- 报告和截图中哪些字段允许对外展示。
- 真实生产 SOP 的 owner 是否允许将发现用于答辩。

Agent 可做：数据格式检查、本地脱敏、风险字段清单、公开版产物生成。

### H2 · 模型访问权和费用上限

用户需提供/拍板：

- 被测外呼模型的 endpoint/访问方式。
- 非 Anthropic 家族模型的可用凭据或平台。
- 本轮实验允许花费的最高额度。
- 允许运行的模型名称和版本。

Agent 可做：配置、小样本试跑、成本预估、受控全量实验、错误恢复与产物对账。

### H3 · 人工质检成本口径

如需宣称 ROI，用户需从业务方获得：

- 人均每日可复核通话数。
- 平均通话时长和单通复核时长。
- 人力成本计算口径。
- 当前抽检率和希望覆盖率。

没有这些数据时，项目只展示自动评测实测成本，不生成绝对节省金额。

## 6. 用户必须亲自做或最终拍板的节点（U）

- [ ] **U1 锁定赛事时间与本轮范围**
  - 需给出：最终截止时间、演示时长、提交形式、是否允许联网/视频/外部 API。
  - 拍板结果：决定做 P0 收口还是连同 P1 增强。
- [ ] **U2 选定跨模型实验集合和预算**
  - 需选：被测模型、裁判模型、每任务样本量、最高成本。
  - Agent 可先提供小样本成本试算，但不擅自开启高成本全量 run。
- [ ] **U3 确认真实数据的使用边界**
  - 需拍板：哪份数据可用、可否外发、脱敏后是否仍敏感、是否需要删除中间产物。
- [ ] **U4 做业务真值裁决**
  - 对编译器黄金集、裁判争议、检查点 severity、“语义等价是否通过”等口径做最终判定。
  - Agent 可预标注和列出争议选项，但不能同时出题、调参和充当最终真值。
- [ ] **U5 持有 sealed holdout**
  - 未公开案例不进入本仓库、不给编写 judge prompt 的 Agent 看。
  - 用户/质检负责人在锁版后运行评分脚本，只回传聚合结果。
- [ ] **U6 拍板 AI 身份披露的合规冲突**
  - 当前真实生产 SOP 存在“被问时否认机器人身份”条款，与安全 policy 的不欺骗原则存在张力。
  - 用户需找业务/合规 owner 拍板：保留红线、分场景处理、还是修 SOP；Agent 不擅自弱化安全门禁。
- [ ] **U7 审美和叙事终审**
  - 用户最终选择首屏话术、品牌感、现场先展示哪个 run、哪些技术细节下沉。
  - Agent 可提供两版候选和理由，但演讲者本人的自然度是最终标准。
- [ ] **U8 授权 Git/部署/官方提交**
  - 本地完成不自动等于可外发。commit、push、更新线上页、发给导师、官方提交需用户当轮明确授权。
- [ ] **U9 亲自完成演讲彩排**
  - 至少完成 3 次计时彩排：正常版、压缩版、无网络版。
  - 记录每次超时点和卡顿点；Agent 据此修文稿，但不能替代用户形成现场口条。
- [ ] **U10 完成官方提交与现场操作**
  - 账号登录、协议确认、官方提交按钮、现场设备和网络准备由用户负责。
  - Agent 可提供提交前清单和现场检查脚本。

## 7. 明确不在本轮实施的事

- 不自己训练大模型。
- 不把评测结果自动回灌并更新外呼模型。
- 不让 Agent 自动修改正式 SOP/checklist/judge prompt 后直接上线。
- 不将对话数据生成的无源检查点并入评分尺。
- 不建设多用户 SaaS 账号/计费/组织后台。
- 不在没有业务真值时宣称确切 ROI。
- 不为显得“Agent 化”而增加无法审计的调度 Agent。
- 不再大量扩充文档或指标，却不补真实对话产品入口。

## 8. 依赖与推荐执行顺序

### 第一批：先让产品真能用

```text
A0-1 → A1-1 → A1-2 → A1-3 → A1-5 → A1-8
                         └→ A1-4
```

```text
A2-1 → A2-5
A2-2

```

收工门：任意报告可回答“谁评的、花了多少、能否与另一次比”。

### 第三批：让报告真能指导修改

```text
A3-1 → A3-2 → A3-3 → A3-4
                    └→ A3-5
                    └→ A3-6
```

收工门：首屏能回答“该改模型、SOP、裁判还是数据，由谁改”。

### 第四批：做成可演示产品

```text
A4-1 → A4-2 → A4-3/A4-4/A4-5 → A4-6 → A4-7
```

收工门：新观众无需解释即能看懂输入、进度和决策输出。

```text
U5-6 由用户保持密封
```

收工门：每个主张都有真实 run、数字和产物定位。

### 第六批：答辩与发布

```text
A6-1 → A6-2/A6-3 → A6-4/A6-5 → H6-6
                                      └→ U7/U9/U10
```

收工门：三分钟不超时，断网可演示，线上与本地同版，官方提交由用户确认完成。

## 10. 整体验收门

所有本轮节点完成必须同时满足：

- 一份新 SOP + 一批已有对话能不经模拟器进入评测。
- 每条正式检查点有 SOP/policy 来源。
- 任一失败可从报告跳转到对话原文和指令来源。
- 报告首屏同时回答门禁、履约、高频问题、根因、owner。
- 裁判中断、NA 过高、尺子不同时 fail-closed。
- 报告记录模型、版本、hash、票数、token、耗时和成本口径。
- 实测/估算/路线图明确区分。
- 旧功能测试不回归；新主链有端到端冒烟。
- 现场可在 3 分钟内完成主故事，失败时可切换缓存资产。
- 任何对外宣称都能在 30 秒内找到原始证据。
