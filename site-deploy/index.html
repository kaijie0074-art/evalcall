为什么这个问题难

美团履约数字人每天产生大规模外呼。指令包含复杂流程分支和多重合规约束，人工抽听无法全量覆盖、标准因人而异。但「做一个自动评测器」不是难点——难的是凭什么信它： 裁判自己会误判，谁来评测评测系统？模型分数低，是模型差还是指令本身写得自相矛盾？模拟器说「测得充分」，充分到什么程度？

EvalCall 的回答是三层质量闭环：不仅评测模型，还审计指令本身，并量化校准自己的裁判。每一层都在官方脱敏数据上实测落地。

第二层 · 审指令Instruction Lint · 评测的不只是模型

实测中任务二被一票否决——深挖发现根因不在模型。EvalCall 内置指令体检器，自动检测指令的自相矛盾 / 不可行约束 / 歧义表述 / 缺失分支，并给出可直接照改的修订建议。以下全部是体检器在两条官方指令上的自主发现（提示词只教方法论，无任何硬编码结论）：

冲突 · high

任务二：字数上限 vs 信息量，物理不可行

指令要求「每次回复极简——最多15-20个字」，但 Step1 参考话术约 43 字、Step2 询问语约 38 字——官方提供的参考话术自己就是上限的两倍以上。任何模型都无法同时满足，实测被一票否决归零。

✏️ 建议：字数上限改为「单句不超过20字、单轮可多句」，或将参考话术拆为 ≤20 字短句分轮发出。

冲突 · high

任务一：业务规则自相矛盾

Call Flow 第 2 步说「单日合同需要连续 Y 天完成配送」，Knowledge Points 却写「单日合同在生效当天完成 X 单即可」——同一合同类型的履约周期在指令内部互相打架，模型说哪个都算错。

✏️ 建议：确认业务实际规则后统一表述；若是笔误，将 Call Flow 第 2 步改为「多日合同需连续 Y 天」。

缺失分支 · high

任务二：没有「请勿再来电」退出分支（合规风险）

整个流程没有商家明确拒绝（「不需要了解」「以后别打了」）时的退出路径。执行者无脚本可循，可能继续硬推，直接引发客诉与合规风险。同时缺失：负责人不在场、说「就1分钟」后仍被拒等路径。

✏️ 建议：补通用退出规则——商家明确拒绝时立即致歉挂断，不得再说任何产品内容。

歧义 · high

任务一：占位符未注入 + 同名变量两种语义

X/Y/Z/W/$ 全部是未填充的占位符且无注入说明，模型只能照念「X单」；其中 Y 在开场白指「每日单数」、在第 2 步又指「连续天数」——同一个变量两种语义。

✏️ 建议：指令顶部增加「系统变量注入」区块，逐一注明变量含义与来源。

两条官方指令各检出 11 条问题，指令可行性分均为 0（确定性公式计算，可复现）。对每天编写外呼指令的运营团队，这是上线前的体检环节：先治指令，再评模型，质检数据才有意义。

第三层 · 校裁判已知答案注入 · 评测结果自带误差棒

LLM 裁判自己会错。EvalCall 用「已知答案注入」校准自己：向系统投放人工构造、真值已知的合成对话（埋好各类违规、补救、未触发分支等难判情形），跑完整裁判流程，输出裁判的查准率/查全率/混淆矩阵。我们把自己裁判的准确率印在报告上——这是评测结果可以被信任的前提。

校准实测：12 条合成对话 · 52 个已知真值（含转述开场白、打断后补救、模糊近违规等难判情形）

指标数值含义
整体准确率86.5%（45/52，宏 F1 0.887）裁判与已知真值的一致程度
违规查全率（fail Recall）100%一个违规都不漏——质检场景最关键的底线
constraint / forbidden / style 准确率100%约束、禁止项、话术类判定高度可信
flow 类准确率82%（误判方向全部为 pass→fail）流程类判定系统性偏严：报告中 flow 类 fail 数自带「约 +18% 偏严」误差棒
违规判定查准率（fail Precision）61.1%（11 TP / 7 FP）裁判报出的违规约四成是偏严误报——我们直接把这个不好看的数字印出来：质检场景宁可错杀不可漏放，但误报需人工复核，这就是误差棒的意义
置信度校准判对均值 0.953 vs 判错均值 0.876裁判判对时确实更自信——置信度可用于可信度分层

误判全部集中在「合理转述的开场白 / 被打断后补救的流程」该不该算合规——这类口径分歧正是校准要暴露给质检负责人拍板的。完整混淆矩阵与逐 case 明细见 runs/calibration/calibration.json。

判定有据：指令原文喂给裁判

裁判提示词携带每条检查点的 source_quote 指令原文。开发中实测：不带原文时裁判对「应当以指定开场白开启」类检查点只能臆测（3 票 2:1 分裂）；带原文后 3 票全对，整体投票一致率 88.9%。

三票多数决 + 分歧率透明

每条 LLM 判定投 3 票取多数，报告同时公布一致率与分裂条目——分裂项自动标记建议人工复核，评测结论自带可信度分层。

上下文隔离

被测模型与模拟器在干净上下文中运行，杜绝宿主环境信息泄入对话轨迹。开发中实测抓到一例轨迹污染（模拟模型拒演并泄露宿主信息），已通过隔离机制根除。

鲁棒 JSON 解析

官方指令含英文引号时，小模型输出 JSON 必带未转义裸引号。内置裸引号修复器（实测失败样本单测覆盖），编译/判定链路对脏输出免疫。

测试充分性量化 · 覆盖率驱动

「模拟器测得充分」不能靠形容词。EvalCall 把每个检查点是否被轨迹实际检验（pass/fail vs na）汇总成触达率，并列出测试盲区清单：

任务检查点触达率测试盲区（从未被任何轨迹触达）
任务一 · 骑手合同通知94.4%1 个分支未触达：「骑手坚持无法配送时应安慰后挂断」——提示需要补一个强硬拒绝型 persona
任务二 · 低延迟直播77.8%6 个条件分支未触达：商家开车场景应挂断、Web控制台未显示选项的兜底引导、被打断时的过渡语等——直接指明下一轮该补哪些 persona/策略

覆盖率不只是事后统计——已实现 coverage-guided 反馈环：每条轨迹判定完成后，仍未触达的检查点被标记为「优先攻击目标」注入下一条轨迹的对抗模拟器，主动制造触发场景（对标 FLARE 的 coverage-guided behavioral fuzzing 范式，evalcall/cli.py 主循环可查证）。评测系统不仅打分，还告诉你测试本身还差什么、并主动去补。

真实战绩：这套系统在开发中抓出的问题

🎯 抓出指令缺陷（第二层）

官方任务二的字数约束 vs 参考话术冲突、任务一的业务规则自相矛盾——若直接上线，质检数据全失真。指令体检在评测前就能拦下。

🎯 抓出裁判误判（第三层）

被测模型一字不差说出指定开场白，单票裁判却判 fail——根因是裁判没拿到指令原文只能臆测。修复后 3 票全对。这正是「校裁判」层存在的意义。

🎯 抓出测试盲区（覆盖率）

任务一有 1 个、任务二有 6 个条件分支从未被测试触达。没有覆盖率量化，这些盲区会被「平均分」掩盖。

学术对标 · 我们站在哪、新在哪

EvalCall 设计对应研究脉络对照结论
指令→检查点编译TICK / CheckEval（ICLR'25）的 checklist 分解评测；Prometheus 2 细粒度 rubric同范式，但我们自动从指令编译且用对抗模拟器动态触发，而非静态评既有文本
规则+LLM 双轨IFEval（可程序化验证约束）+ MT-Bench 系 LLM-judgeIFEval 只覆盖可正则验证的约束，我们把语义约束一并纳入
对抗式用户模拟器τ-bench / τ²-bench（NeurIPS'24，Sierra）的 LLM 用户模拟τ 系列追求「像真人」（fidelity），我们追求「专挑指令崩溃路径」（adversarial coverage）
裁判自校准JudgeBench（ICLR'25）的 meta-evaluation with verifiable ground truth同方法论，落到外呼质检场景
覆盖率反馈环FLARE 的 coverage-guided behavioral fuzzing（此前只用于代码/MAS 测试）首次用于对话指令遵循质检
指令体检器——公开文献空白：IFEval/ComplexBench/FollowBench 都假设指令是好的、只测模型；没有公开工作检测「指令本身是否自相矛盾/不可行/缺分支」

一句话定位：把 IFEval 的可验证约束、τ-bench 的用户模拟、JudgeBench 的裁判元评测，首次合成一个面向外呼 agent 的上线前主动压测闭环——并新增「指令体检」这个文献空白维度。「对抗式 × 覆盖率引导 × 外呼指令遵循」三元交集查无公开先例。

生产成本账 · 比人工省多少

项目测算
单条轨迹 LLM 调用量≈35 次（被测 10 + 模拟器 10 + 裁判 5 批×3 票），约 8 万 token
生产后端单价（如 qwen-plus / deepseek 档）约 ¥0.2–0.6 / 轨迹；规则轨零成本前置过滤可再降 LLM 调用 30%+
人工质检对照抽听一通电话+填表约 10–15 分钟，按质检员时薪折算 ≈¥8–12 / 通
结论单通成本约为人工的 1/20–1/50，且可全量覆盖；裁判偏严带来的误报（fail 查准率 61%）意味着约 4 成 fail 需人工复核——复核量仍远小于全量抽听

诚实声明：本演示用本地 claude-cli 后端（单次 30-180s），慢是演示通道的选择，不是架构上限——后端可插拔，切换批量 API 即得生产吞吐。裁判升级后可用重判工具从既有轨迹回溯，无需重跑对话，回归成本约为全量的 1/3。

已知局限与路线图 · 我们自己先说

同家族裁判的自我偏袒

当前被测与裁判同为 Claude 家族，存在 self-enhancement bias。已实现跨档位裁判团（haiku/sonnet/opus 每票不同模型），生产部署将换异构家族（GPT/Qwen/Claude 交叉）+ 人工抽检锚点。

编译器本身尚未校准

检查点编译靠 source_quote 溯源兜底，但「抽得全不全、severity 标得准不准」无金标准校验。路线图：把校准黄金集扩展到编译层（人工标注的标准检查点清单）。

样本规模与统计置信

演示规模下分数为观测值而非统计估计。已支持多轨迹 + pass^k 可靠性口径（对标 τ-bench），生产建议每 persona ≥20 条轨迹。

生态效度

被测模型为通用 LLM 扮演（官方未提供真实外呼模型，所有参赛队同此约束）；persona 为手工设计+盲区驱动补充。真实脱敏轨迹接入有现成格式适配层。

使用方式

evalcall run --task task.yaml --personas all　编译指令 → 对抗模拟 → 双轨判定

evalcall report --run runs/xxx　生成可交互 HTML 报告

python3 calibrate.py　裁判自校准：已知答案注入 → 查准率 / 查全率

python3 rejudge_checkpoint.py runs/xxx all 3　裁判升级后从既有轨迹全量重判（无需重跑对话）
