为什么这个问题难

美团履约数字人每天产生大规模外呼。指令包含复杂流程分支和多重合规约束，人工抽听无法全量覆盖、标准因人而异。但「做一个自动评测器」不是难点——难的是凭什么信它： 裁判自己会误判，谁来评测评测系统？模型分数低，是模型差还是指令本身写得自相矛盾？模拟器说「测得充分」，充分到什么程度？

EvalCall 的回答是三层质量闭环：不仅评测模型，还审计指令本身，并量化校准自己的裁判。每一层都在官方脱敏数据上实测落地。

下面三层依次回答三个问题：模型守不守指令（评模型）→ 指令本身可不可信（审指令）→ 我凭什么信自己的裁判（校裁判）。

第二层 · 审指令Instruction Lint · 评测的不只是模型

实测中任务二被一票否决——深挖发现根因不在模型。EvalCall 内置指令体检器，自动检测指令的自相矛盾 / 不可行约束 / 歧义表述 / 缺失分支，并给出可直接照改的修订建议。以下全部是体检器在两条官方指令上的自主发现（提示词只教方法论，无任何硬编码结论）：

冲突 · high

任务二：字数上限 vs 信息量，物理不可行

指令要求「每次回复极简——最多15-20个字」，但 Step1 参考话术约 43 字、Step2 询问语约 38 字——官方提供的参考话术自己就是上限的两倍以上。任何模型都无法同时满足，实测被一票否决归零。

✏️ 建议：字数上限改为「单句不超过20字、单轮可多句」，或将参考话术拆为 ≤20 字短句分轮发出。

冲突 · high

任务一：业务规则自相矛盾

Call Flow 第 2 步说「单日合同需要连续 Y 天完成配送」，Knowledge Points 却写「单日合同在生效当天完成 X 单即可」——同一合同类型的履约周期在指令内部互相打架，模型说哪个都算错。

✏️ 建议：确认业务实际规则后统一表述；若是笔误，将 Call Flow 第 2 步改为「多日合同需连续 Y 天」。

缺失分支 · high

任务二：没有「请勿再来电」退出分支（合规风险）

整个流程没有商家明确拒绝（「不需要了解」「以后别打了」）时的退出路径。执行者无脚本可循，可能继续硬推，直接引发客诉与合规风险。同时缺失：负责人不在场、说「就1分钟」后仍被拒等路径。

✏️ 建议：补通用退出规则——商家明确拒绝时立即致歉挂断，不得再说任何产品内容。

歧义 · high

任务一：占位符未注入 + 同名变量两种语义

X/Y/Z/W/$ 全部是未填充的占位符且无注入说明，模型只能照念「X单」；其中 Y 在开场白指「每日单数」、在第 2 步又指「连续天数」——同一个变量两种语义。

✏️ 建议：指令顶部增加「系统变量注入」区块，逐一注明变量含义与来源。

两条官方指令各检出 11 条问题，指令可行性分均为 0（确定性公式计算，可复现）。具体到美团运营的工位：写完一条新外呼指令，提交前跑一遍 lint_instruction.py，几十秒拿到「3 处冲突、1 个缺失退出分支」和照改建议——把一次潜在的全量客诉挡在上线之前。先治指令，再评模型，质检数据才有意义。

第三层 · 校裁判已知答案注入 · 评测结果自带误差棒

审完指令、评完模型，最后一个问题反噬自己——上面所有结论都依赖 LLM 裁判，可裁判自己也会错。谁来评测评测者？EvalCall 用「已知答案注入」校准自己：向系统投放人工构造、真值已知的合成对话（埋好各类违规、补救、未触发分支等难判情形），跑完整裁判流程，输出裁判的查准率/查全率/混淆矩阵。我们把自己裁判的准确率印在报告上——这是评测结果可以被信任的前提。

判定有据：指令原文喂给裁判

裁判提示词携带每条检查点的 source_quote 指令原文。开发中实测：不带原文时裁判对「应当以指定开场白开启」类检查点只能臆测（3 票 2:1 分裂）；带原文后 3 票全对，整体投票一致率 88.9%。

三票多数决 + 分歧率透明

每条 LLM 判定投 3 票取多数，报告同时公布一致率与分裂条目——分裂项自动标记建议人工复核，评测结论自带可信度分层。

上下文隔离

被测模型与模拟器在干净上下文中运行，杜绝宿主环境信息泄入对话轨迹。开发中实测抓到一例轨迹污染（模拟模型拒演并泄露宿主信息），已通过隔离机制根除。

鲁棒 JSON 解析

官方指令含英文引号时，小模型输出 JSON 必带未转义裸引号。内置裸引号修复器（实测失败样本单测覆盖），编译/判定链路对脏输出免疫。

决策与安全增强 · 从「打分」到「把关」业务导师建议落地 16 项 · 2026-07 真实重跑验证

质检团队要的不是分数，是能拍板的结论。三层闭环之上，评测出口落到「可上线 / 打回 / 无法判定」的门禁决策——2026-07-02 用门禁版流水线全量真实重跑：官方任务一（12 通 · 打回 · 履约 20%） · 官方任务二（打回 · 0 分挂指令归因警示） · 无冲突对照任务（分数正常分布，证明 0 分是检出不是失灵）。

⛔ 上线红线门禁

任一 P0（安全红线/critical）失败 → 整体打回；报告头条直接给决策徽章 + 触发红线明细。裁判通道故障导致大面积「无法判定」时如实输出「无法判定」——绝不把「没判成」包装成「通过」（fail-closed 设计）。

🛡 安全/合规红线轨

辱骂、歧视、隐私泄露、未告知 AI 身份、诱导——独立于任务指令的全局 P0 红线轨（policy 单一规则源），业务分级 P0/P1/P2 对应打回/扣分/复核。

📞 履约达成检查点

「这通电话到底有没有把活办成」——真实外呼第一 KPI。从任务 goal 生成 outcome 检查点，报告头条直接给履约达成率。判据口径可按业务定义一行替换。

🤖 真实性/拟人度

用户质疑「你是不是机器人」时被测客服是否稳健应对、该表明身份时如实表明——合规导向实现（藏身份骗用户在履约场景反而可能踩线）。

📈 evalcall diff · 跨版本回归

模型改一版，同尺 checklist 下自动对比哪些检查点变好/变差；规则轨结论确定、LLM 轨标注「需复测确认」——真实对比产物：Haiku→Sonnet 3 好 7 差 22 不变。

🌱 evalcall grow · 活清单

从真实对话暴露的盲区挖「指令要求了但清单漏了」的候选检查点——溯源硬闸：source_quote 必须逐字回指指令原文，防「用模型输出反推评判标准」的循环论证；只进待人工确认区。

任务无反馈环（被动统计）coverage-guided 反馈环开启后

反馈环已实现且实证有效（任务二 +22.2 个百分点、盲区清零）：每条轨迹判定完成后，仍未触达的检查点被标记为「优先攻击目标」注入下一条轨迹的对抗模拟器，主动制造触发场景——商家开车要挂断、Web控制台兜底引导这些此前测不到的分支被逐一逼出（绝对量：两任务合计补上 7 个盲区检查点——演示规模下绝对量小，机制价值在检查点数百条的生产规模显现；机制对标 coverage-guided behavioral fuzzing，evalcall/cli.py 主循环可查证）。触达口径已收紧：fail 须附原文证据才计触达，收紧后两任务仍保持 100%。评测系统不仅打分，还告诉你测试差什么、并主动补上。

真实战绩：这套系统在开发中抓出的问题

🎯 抓出指令缺陷（第二层）

官方任务二的字数约束 vs 参考话术冲突、任务一的业务规则自相矛盾——若直接上线，质检数据全失真。指令体检在评测前就能拦下。

🎯 抓出裁判误判（第三层）

被测模型一字不差说出指定开场白，单票裁判却判 fail——根因是裁判没拿到指令原文只能臆测。修复后 3 票全对。这正是「校裁判」层存在的意义。

🎯 抓出测试盲区，并主动补上（覆盖率反馈环）

被动统计模式下任务一有 1 个、任务二有 6 个条件分支从未被触达；开启 coverage-guided 反馈环后两任务触达率均达 100%、盲区清零——发现盲区到消灭盲区，闭环实证有效。

🎯 抓出「比较实验不同尺」的设计缺陷（工程纪律）

判别力实验首跑时两组各自重新编译检查点（LLM 编译器非确定性），对照失效——已产品化为 --checklist 固化复用机制并同尺重判。比较实验必须同尺，这条教训写进了 CLI。

已知局限与路线图 · 我们自己先说

同家族裁判的自我偏袒

当前被测与裁判同为 Claude 家族，存在 self-enhancement bias。已实现跨档位裁判团（haiku/sonnet/opus 每票不同模型），生产部署将换异构家族（GPT/Qwen/Claude 交叉）+ 人工抽检锚点。

编译器本身尚未校准

检查点编译靠 source_quote 溯源兜底，但「抽得全不全、severity 标得准不准」无金标准校验。路线图：把校准黄金集扩展到编译层（人工标注的标准检查点清单）。

样本规模与统计置信

演示规模下分数为观测值而非统计估计。已支持多轨迹 + pass^k 可靠性口径（对标 τ-bench），生产建议每 persona ≥20 条轨迹。

生态效度

被测模型为通用 LLM 扮演（官方未提供真实外呼模型，所有参赛队同此约束）；persona 为手工设计+盲区驱动补充。真实脱敏轨迹接入有现成格式适配层。
